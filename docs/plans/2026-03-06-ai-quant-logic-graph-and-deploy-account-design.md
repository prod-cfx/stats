# AI量化逻辑图确认层与部署账户选择设计

## 1. 背景与目标
- 背景：当前 AI量化链路是“自然语言 -> 回测 -> 部署”，缺少用户可视确认层，且部署时未显式选择交易所账户。
- 目标：增加“逻辑图确认闸门”与“部署账户选择闸门”，确保用户先确认策略语义，再消耗回测资源，再明确部署账户。

## 2. 核心原则（已确认）
- 逻辑图是给用户确认是否正确，不是 AI 自动判定可不可。
- 不允许强制回测：未确认逻辑图时禁止回测。
- 采用方案A：双闸门流程。

## 3. 主流程（方案A）
1. 用户输入自然语言。
2. AI 输出：
   - 文本解释
   - 逻辑图（只读）
3. 进入待确认状态：
   - 主按钮：`确认并回测`
   - 次按钮：`返回对话修改`
   - `开始回测` 按钮禁用
4. 用户点击 `确认并回测` 后才触发回测。
5. 回测通过后进入部署弹窗：必须选择 `交易所 + API账户`。
6. 部署成功后写入个人中心策略记录（运行中）。

## 4. 部署账户模型
### 4.1 账户级 API 绑定（个人中心）
每个 API 账户记录包含：
- `accountId`
- `exchange`（binance/okx）
- `accountName`（用户命名）
- `apiKeyMask`
- `status`（available/invalid）

### 4.2 部署弹窗硬约束
部署按钮可点前提：
- 逻辑图已确认
- 回测达标
- 已选择交易所
- 已选择该交易所下可用账户

若无可用账户：展示引导跳转到 `账号中心 -> AI量化 -> API配置`。

### 4.3 部署记录
策略记录新增：
- `deploy.exchange`
- `deploy.accountId`
- `deploy.accountName`
- `deploy.at`
- `deploy.status`

## 5. 逻辑图中间结构与展示
### 5.1 数据结构
- `trigger`：触发条件组（AND/OR）
- `actions`：执行动作组（buy/sell/close/notify）
- `risk`：风控组（止盈/止损/仓位上限）
- `meta`：交易所、交易对、周期、仓位、策略名
- `version`：逻辑图版本号

### 5.2 UI 规范（只读）
- 分层块：`IF` / `THEN` / `AND|OR` / `EXECUTE`
- 支持折叠，不支持拖拽与编辑
- 每次 AI 重生成都产生新版本

### 5.3 版本与确认状态
- `draft`：未确认
- `confirmed`：已确认
- 用户修改自然语言后，旧版本确认失效，新版本回到 `draft`

## 6. 异常处理
- 逻辑图生成失败：提示“生成失败，请重试”，不触发回测。
- 字段缺失：展示待补参数清单，提供“继续对话补全”。
- 无可用账户：部署按钮禁用并引导配置。

## 7. 验收标准（MVP）
1. 自然语言提交后先出现逻辑图，不直接回测。
2. 未确认逻辑图时回测入口不可用。
3. 点击“确认并回测”后才真正触发回测。
4. 部署时必须选择“交易所+账户”。
5. 部署成功后个人中心策略详情可看到部署账户信息。
6. 全链路可在前端 mock 下跑通。

## 8. 本轮边界
- 逻辑图仅只读，不做拖拽编辑。
- 不接真实交易执行后端。
- 以 UI 流程与状态约束为主。
