# AI-Quant 回测页补齐设计（历史回测区间）

日期：2026-03-24  
范围：`apps/front` AI-Quant 前端页面  
目标：在现有回测流程中补齐“历史回测区间”能力（快捷区间 + 自定义起止时间），并保证状态持久化与错误处理完整。

## 1. 背景与现状

已确认现状：
- 回测详情页是占位实现，核心指标基于 `id` 派生，并非真实回测输入驱动。
- 参数面板仅覆盖 `exchange/symbol/positionPct`，未提供历史回测区间。
- 用户目标明确为：新增历史回测区间，输入形式为“快捷区间（7D/30D/90D/1Y）+ 可选自定义开始/结束时间”。

## 2. 需求拆解

### 2.1 显式需求
- 支持快捷区间：`7D`、`30D`、`90D`、`1Y`。
- 支持切换到“自定义”，填写 `开始时间` 和 `结束时间`。
- 回测执行使用统一解析后的时间范围。
- 回测结果可追溯展示使用的区间。
- 刷新后保留区间配置（沿用会话本地存储）。

### 2.2 隐含需求
- 不破坏现有 `exchange/symbol/positionPct` 使用方式。
- 不引入大规模重构（KISS/YAGNI）。
- 对旧本地会话数据兼容迁移。

## 3. 方案对比与结论

### 方案 A（推荐）
在现有 `QuantParams` 上扩展回测区间字段，参数面板直接增加区间交互，回测请求统一携带解析后区间。

优点：
- 改动集中、实现快、回归风险低。
- 复用当前聊天页回测入口与会话状态。

缺点：
- 回测详情页短期仍以轻量展示为主，不做完整重构。

### 方案 B
回测区间只在详情页做完整筛选与二次回测，聊天页保留简单配置。

缺点：
- 状态同步复杂，易产生两套参数源。

### 方案 C
重做独立回测流程页（配置→执行→结果）。

缺点：
- 周期长，超出本次需求边界。

结论：采用方案 A。

## 4. 详细设计

### 4.1 数据模型
在 `QuantParams` 增加：
- `backtestRangePreset`: `'7D' | '30D' | '90D' | '1Y' | 'CUSTOM'`
- `backtestStart`: `string`（ISO 或本地可解析时间串）
- `backtestEnd`: `string`

默认值：
- 预设 `30D`
- `end = now`
- `start = now - 30D`

旧会话兼容：
- 若缺失新字段，初始化为上述默认值后继续渲染。

### 4.2 交互设计
参数面板新增“历史区间”区域：
- 快捷按钮：`7D/30D/90D/1Y/自定义`
- 当选择 `CUSTOM` 时展示两个 `datetime-local` 输入框（开始/结束）

交互约束：
- 选择快捷区间时自动计算 `start/end`。
- 切换 `symbol`、`exchange` 不重置区间。
- 仅在输入合法时允许触发回测。

### 4.3 回测执行数据流
点击“开始回测”时：
1. 前端校验区间合法性。
2. 归一化出最终 `startAt/endAt`。
3. 回测执行逻辑读取统一参数对象（含区间字段）。
4. 回测结果区域显示本次区间，确保可追溯。

### 4.4 详情页最小改造
`backtest/[id]` 页不再只依赖 `id` 随机派生视图。
MVP 最小要求：
- 至少展示本次回测的 `symbol` 与 `startAt/endAt`。
- 指标可暂时沿用 mock 逻辑，但输入参数来源必须真实。

## 5. 错误处理

前端阻断条件：
- `start/end` 为空：提示“请完整选择历史区间”。
- `start >= end`：提示“开始时间必须早于结束时间”。
- 跨度超限（建议 365 天）：提示“区间过大，请缩小范围”。

回测失败：
- 保留用户输入参数。
- 展示可重试错误提示。

## 6. 测试清单（需求驱动）

1. 快捷区间 `7D/30D/90D/1Y` 生成正确 `start/end` 并可触发回测。
2. 自定义区间合法时可回测，并在结果区显示正确区间。
3. 边界：空值、`start=end`、`start>end`、跨度 365 天、跨度 366 天。
4. 异常：接口失败后参数不丢失，重试路径可用。
5. 状态：刷新后区间保留；旧 localStorage 数据自动补齐默认值。
