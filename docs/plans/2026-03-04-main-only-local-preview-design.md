# Main-Only 本地预览设计（防止已合并功能“看起来丢失”）

## 背景与问题
用户系统与监控功能已分别通过 PR 合并到 `main`，但本地演示时若从 feature 分支启动，会看到分支中间态，导致“登录系统没了/功能不一致”的假象。

## 目标
- 所有“部署本地看效果”场景只展示 `main` 的已合并状态。
- 演示流程可重复、可核对、可追责。
- 不改变研发分支开发方式。

## 方案选择
已确认采用：`方案1 Main-Only 本地演示`。

## 设计
### 1) 分支准入规则
- 触发词：当请求包含“部署本地 / 看效果 / 本地预览 / 演示”。
- 强制规则：只能在 `main` 分支启动服务。
- 若当前不在 `main`：先停止并提示，切换到 `main` 后继续。

### 2) 标准执行流程
1. `git checkout main`
2. `git pull --ff-only`
3. 启动服务（front/backend）
4. 回报三项信息：
   - 当前分支（必须是 `main`）
   - 当前 commit short sha
   - 访问地址

### 3) 验收与证据
- 验收标准：用户可在本地看到与 `main` 一致的登录与监控功能。
- 证据要求：回复中必须包含 `branch + sha + url`，不再只报 URL。

### 4) 失败策略
- `main` 拉取失败：停止演示并反馈失败原因，不切回 feature 分支继续演示。
- 端口冲突：仅调整端口，不改变分支规则。

## 非目标
- 不解决后端 API 协议差异。
- 不引入新脚本或 CI（后续可升级）。

## 运行手册（最小版）
```bash
git checkout main
git pull --ff-only
# start services
```

回复模板：
- `branch: main`
- `sha: <short-sha>`
- `url: http://localhost:<port>/zh/...`
