# AI Quant 对话流畅性修复设计（短确认词自动推进）

日期：2026-03-23
状态：已评审通过（用户确认）

## 1. 问题定义
当前 AI Quant 对话在用户输入短确认词（如“可以”“继续”“按你说的来”）时，常重复追问同一槽位（尤其入场逻辑），出现“上下文不流畅、像状态机卡住”的体验问题。

典型表现：
1. assistant 已给出明确策略草案；
2. 用户回复确认词；
3. assistant 仍重复“请具体描述入场逻辑”，形成循环。

## 2. 目标
1. 用户在草案阶段回复短确认词时，默认接受上一条草案。
2. 自动确认逻辑图并继续生成代码（保留安全闸门语义）。
3. 消除同一槽位重复追问，避免对话死循环。
4. 不破坏非确认场景下的正常补问能力。

## 3. 非目标
1. 不重构整体 LLM 对话架构。
2. 不修改策略协议（StrategyAdapterV1/StrategyDecisionV1）与回测/实盘执行协议。
3. 不引入新的后端状态机模型（本轮优先前端止血）。

## 4. 方案对比

### 方案 A（选中）前端语义路由增强
- 识别“短确认词 + 已有草案上下文”，直接走 `confirmGenerate=true`。
- 本地将 `logicGraph.status` 置为 `confirmed`，并推进生成链路。

优点：
1. 改动范围小，发布快。
2. 可精准修复当前循环路径。

缺点：
1. 需要维护短确认词与草案特征规则。

### 方案 B 后端编排器容错增强
- 后端在 `currentLogic` 完整且用户短确认时直接 `logicReady=true`。

### 方案 C 前后端双保险
- A + B 同时实施。

结论：先上 A 快速止血，后续再评估补 B 做兜底。

## 5. 详细设计

### 5.1 触发规则（已确认）
当满足以下条件时，触发“自动推进生成”：
1. 用户输入短确认词（`可以/确认/继续/按你说的来/ok/yes` 等）；
2. 当前会话存在 `logicGraph`；
3. 最近一条 assistant 消息包含策略草案特征（如“入场条件/出场条件/策略逻辑如下/请确认逻辑图”）。

触发后行为：
1. `logicGraph.status=confirmed`；
2. 直接调用生成路径（`confirmGenerate=true`）；
3. 禁止再次进入“请具体描述入场逻辑”的重复追问。

### 5.2 防重复追问机制（已确认）
新增“追问指纹”控制：
1. 指纹 = `槽位类型(entry/exit/risk) + 问句模板`；
2. 若短确认触发自动推进，跳过追问态；
3. 若非短确认且字段缺失，允许追问一次；
4. 连续两轮同指纹追问时，自动给默认草案并进入确认生成分支，避免卡死。

### 5.3 文案与状态一致性
1. 自动推进后文案应体现“已根据已确认逻辑继续生成”。
2. 禁止出现“请重新描述核心逻辑/我们重新开始”等回退感文案。
3. 当会话终态重建时，必须复用已确认逻辑图 checklist。

## 6. 测试策略

### 6.1 前端测试（必做）
1. `短确认词 + 有logicGraph + assistant草案` => 自动 `confirmGenerate=true`。
2. 同场景下不会再出现“请具体描述入场逻辑”。
3. 连续短确认词（“可以”→“继续”）不会卡在重复问句。
4. 非短确认词保持原补问流程。

### 6.2 联调验收（必做）
1. 复现当前问题路径，确认不再循环追问。
2. 终态重建 session 后，仍沿用当前逻辑图 checklist。
3. 生成链路可连续推进至代码产出。

## 7. 风险与缓解
1. 风险：短确认词规则误判。
   - 缓解：限定需同时满足“草案上下文 + logicGraph 存在”。
2. 风险：过度自动推进导致跳过必要补充。
   - 缓解：仅在草案完整特征命中时自动推进，其余保持补问。
3. 风险：前端规则与后端规划器语义漂移。
   - 缓解：后续补后端兜底（方案 B）。

## 8. DoD
1. 截图场景不再出现重复“请具体描述入场逻辑”。
2. 用户输入“可以/继续”可顺畅推进至生成代码。
3. 前端测试与联调验收通过。
