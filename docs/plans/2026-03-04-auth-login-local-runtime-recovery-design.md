# Auth 登录本地运行恢复设计（不改代码）

## 背景
用户反馈：
1. 登录页在本地看不到预期样式（尤其白天模式字体观感）。
2. 邮箱验证码登录报错（截图显示 `HTTP_500`）。

已确认：
- `PR #399/#402/#404/#405` 已合并到 `main`。
- `http://localhost:3011` 前端默认通过 `/api/v1/*` 代理到 `http://localhost:3000`。
- 当前问题主要是本地运行态：后端未在 `3000` 可用时，OTP 登录必然失败。

## 目标
- 在不修改任何业务代码的前提下，恢复本地 OTP 登录可用性。
- 使用已合并到 `main` 的真实代码与样式进行验收。
- 输出可复核证据，避免“看起来像代码丢失”的误判。

## 非目标
- 不改登录 UI 代码。
- 不改前端 auth API 代码。
- 不改后端业务接口实现。

## 方案对比
### 方案 A（推荐）：运行态恢复（Main + Backend 3000）
- 做法：
  - 从 `main` 启动 front。
  - 启动 backend 到 `3000`。
  - 验证 OTP 发码与登录流程。
- 优点：不引入代码风险，直接验证“你已做好的实现”。
- 缺点：依赖本地后端与邮件配置可用。

### 方案 B：前端改地址到其他后端
- 做法：修改本地运行环境变量，让 front 指向别的后端地址。
- 优点：可复用已有远端后端。
- 缺点：偏离本地标准链路，不利于统一排查。

### 方案 C：前端增加伪成功兜底（不推荐）
- 做法：接口失败时本地假成功。
- 缺点：状态不一致，误导验收。

## 采用方案
采用 **方案 A**。

## 详细设计
### 1) 启动与路由基线
- Front：`main` 分支启动（避免 feature 中间态）。
- Back：监听 `3000`。
- 代理：front 的 `/api/v1/:path*` -> `http://localhost:3000/api/v1/:path*`。

### 2) OTP 验收流程
- 打开 `http://localhost:3011/zh/auth/login`。
- 输入邮箱，点击发送验证码。
- 收到成功响应后输入验证码并登录。
- 登录后跳转账户页或出现 authenticated 状态。

### 3) 样式验收口径
- 在白天模式下确认：
  - 标题、输入框、按钮、错误文案的对比度与 Coinflux 视觉 token 一致。
  - 不以 feature 分支截图为准，以 `main` 合并后的实际渲染为准。

### 4) 证据输出规范
每次本地验收必须输出：
- `front branch`
- `front sha`
- `backend port`
- `login page url`
- `OTP send/verify 结果`

## 风险与回滚
- 风险：3000 被占用或 backend env 未加载。
- 处理：使用标准命令启动 backend（自动加载 env），若端口冲突先释放端口。
- 回滚：仅停止进程，不改代码文件。
