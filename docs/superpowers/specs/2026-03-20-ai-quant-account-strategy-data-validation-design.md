# AI 量化账户页策略数据正确性验证设计

- 日期：2026-03-20
- 范围：`apps/quantify` 账户策略查询接口 + `apps/front` 账户页策略列表/详情数据映射与展示
- 目标：先产出完整测试方案和用例清单，确认后再决定是否改代码

## 1. 背景与问题

当前需要验证 AI 量化账户页两类页面的数据是否正确：

1. 策略列表页
2. 策略详情页

已确认的验收口径：

1. 既要校验后端返回，也要校验前端二次处理结果（双重校验）
2. `总收益额` 的业务口径固定为纯收益 `PnL`，不是本金 + PnL

现状关键点（来自现有代码）：

1. 大部分核心指标由后端聚合返回（如收益率、最大回撤、胜率、交易次数）。
2. 前端对 `totalPnl` / `todayPnl` 存在 fallback 计算路径。
3. fallback 路径里存在潜在口径漂移风险：名称是“总收益额”，但兜底值来源接近“总资产”。

## 2. 设计目标

1. 建立可重复执行的分层验证方案，覆盖接口、映射、页面展示三层。
2. 让每个失败都能快速定位到具体层级（后端口径、前端适配、组件展示）。
3. 对“总收益额=纯收益(PnL)”建立明确测试门禁，避免后续回归。
4. 在不引入过度工程的前提下，保留少量页面级烟测保证用户视角。

## 3. 方案选型

### 方案 A：人工核对优先

- 做法：抓包 + 页面截图 + 手工算公式。
- 优点：启动快。
- 缺点：不可回归，易漏测，难定位。

### 方案 B：分层自动化（采用）

- 做法：接口层 + 适配层 + 组件层为主，补最小页面烟测。
- 优点：定位快、可回归、维护成本可控。
- 缺点：初次整理样本与断言需要一次性投入。

### 方案 C：全链路 E2E 优先

- 做法：端到端覆盖所有口径校验。
- 优点：最贴近真实用户路径。
- 缺点：执行慢、脆弱、失败定位成本高。

结论：采用方案 B，方案 C 仅保留 1 条最小烟测。

## 4. 测试范围与非目标

### 4.1 In Scope

1. 列表接口数据与列表页展示一致性。
2. 详情接口数据与详情页展示一致性。
3. API DTO 到前端页面模型映射的字段正确性。
4. fallback 触发条件与公式正确性。
5. 策略 run/stop 操作后的列表与详情收敛一致性。

### 4.2 Out of Scope

1. 量化策略交易逻辑正确性（撮合、下单、风控本身）。
2. 回测引擎准确性。
3. 非账户页的其他 AI 量化页面。

## 5. 数据口径与判定规则

### 5.1 单一判定基准

1. 若后端字段存在：页面必须展示后端值（允许格式化差异，如小数位、千分位、正负号）。
2. 若后端字段缺失：才允许走前端 fallback。
3. 优先级固定：`后端字段` > `fallback 计算值` > `默认值 0`。
4. 缺失判定：仅 `null` 或 `undefined` 视为缺失。
5. 有效值判定：`0` 是有效值，不触发 fallback。
6. 非法数值判定：`NaN` 或非有限数值视为无效，按缺失处理，允许触发 fallback。

### 5.2 关键口径

1. `总收益额`：必须是纯收益 `PnL`。
2. `今日盈亏`：必须是当日盈亏，不得混入累计收益。
3. `收益率/最大回撤/胜率/交易次数`：以后端返回为准。

### 5.3 状态映射

1. 后端 `paused` 在账户 UI 视图中映射为 `stopped`。
2. 接口契约期望：账户页聚合接口对外只返回 `running|stopped|draft`，不直接暴露 `paused`。
3. 前端适配层仍保留防御式映射：非 `running|draft` 统一落到 `stopped`。

### 5.4 fallback 公式（可执行定义）

1. `displayTotalPnl`（总收益额）：
- 当 `totalPnl` 非空时：直接使用 `totalPnl`。
- 当 `totalPnl` 为空时：
 - 若 `equitySeries.length >= 2`：`totalPnl = last(equitySeries.value) - first(equitySeries.value)`。
 - 若 `equitySeries.length = 1` 且存在 `initialCapital`：`totalPnl = onlyPoint.value - initialCapital`。
 - 其他情况：`totalPnl = 0`。
2. `displayTodayPnl`（今日盈亏）：
- 当 `todayPnl` 非空时：直接使用 `todayPnl`。
- 当 `todayPnl` 为空时：
 - 取 `equitySeries` 最后一个点所在自然日 `D`。
 - 从 `equitySeries` 过滤同日 `D` 的点集 `daySeries`。
 - 若 `daySeries.length >= 2`：`todayPnl = last(daySeries.value) - first(daySeries.value)`。
 - 其他情况：`todayPnl = 0`。
3. 数值规则：
- 精度：统一四舍五入到小数点后 2 位。
- 符号：保留正负；展示层可加 `+` 前缀但不改变数值语义。
- 单位：`USDT`。
4. 时间规则（用于 `todayPnl`）：
- 计算“自然日 D”时，统一使用 `UTC` 日界（`00:00:00Z` 到 `23:59:59Z`）。
- 页面可按本地时区格式化展示时间文本，但断言层按 `UTC` 计算。

## 6. 测试数据准备

## 6.1 账号准备

1. `userA`：策略创建者。
2. `userB`：订阅者（用于权限与视角差异）。

## 6.2 策略样本准备（最少 4 条）

1. `S1`：`running`，指标完整，`totalPnl/todayPnl` 均存在。
2. `S2`：`stopped`，`todayPnl` 缺失（验证 fallback）。
3. `S3`：`draft`，部分字段为 `null`（验证前端归一和展示占位）。
4. `S4`：存在 `equitySeries` 但 `totalPnl` 缺失（验证“总收益额”fallback 口径）。

## 6.4 样本真值表（最小）

1. `S1`（完整字段）：
- 后端：`status=running`，`totalPnl=320.12`，`todayPnl=18.34`
- 预期：`displayTotalPnl=320.12`，`displayTodayPnl=18.34`
- 预期：不触发任何 fallback。
1. `S2`（`todayPnl` 缺失）：
- `equitySeries`（同日）：`[10000, 10080, 10050]`
- 预期：`displayTodayPnl = 50.00`
- 若后端给 `totalPnl=320.12`，则 `displayTotalPnl = 320.12`（不走 fallback）
1. `S3`（空值与状态映射）：
- 后端：`status=paused`（内部态），账户聚合输出应为 `stopped`
- 后端：`totalPnl=0`，`todayPnl=0`
- 预期：`displayTotalPnl=0.00`，`displayTodayPnl=0.00`
- 预期：`0` 作为有效值，不触发 fallback。
2. `S4`（`totalPnl` 缺失）：
- `equitySeries`：`[10000, 10120, 10070]`
- `initialCapital=10000`
- 预期：`displayTotalPnl = 70.00`（纯收益，不是 10070.00）
- 若同日点为 `10020 -> 10070`，预期：`displayTodayPnl = 50.00`

## 7. 用例矩阵（分层）

## 7.1 接口层（Contract/Service）

1. 列表接口：字段完整性与类型正确性。
2. 列表接口：状态映射前置值覆盖（含 `paused -> stopped`）。
3. 详情接口：`metrics` 与 `totalPnl/todayPnl/equitySeries/timeline/snapshot` 输出一致性。
4. 详情接口：空数据与缺失字段行为。
5. 动作接口：run/stop 后返回详情与后续查询一致。

## 7.2 前端适配层（API Adapter）

1. 列表项映射：字段透传、默认值、状态转换。
2. 详情映射：`totalPnl/todayPnl` 透传优先。
3. 空值归一：`null/undefined/NaN` 到 UI model 的稳定行为。
4. 时间字段格式转换与可读性。

## 7.3 组件层（List + Detail）

1. 列表页：
 - 展示字段与接口一致。
 - run/stop 操作后刷新一致。
2. 详情页：
 - 指标卡按后端值展示。
 - `totalPnl` 存在时展示纯收益值。
 - `totalPnl` 缺失时，fallback 仍输出纯收益口径。
 - `todayPnl` 缺失时 fallback 正确。
  - `paused` 不在详情 UI 出现，映射后仅见 `stopped`。

## 7.4 页面烟测（最小 E2E）

1. 从列表进入详情，关键指标在两个页面与接口保持一致。
2. 触发一次 run/stop，确认列表状态与详情状态同步收敛。

## 8. 执行顺序

1. 先固定样本数据与预期真值表（source of truth）。
2. 执行接口层测试，确认后端口径稳定。
3. 执行适配层测试，确认映射和默认值稳定。
4. 执行组件层测试，确认展示优先级和 fallback 公式稳定。
5. 最后跑最小页面烟测，确认用户路径无口径漂移。

## 9. 通过标准

1. 同一策略在“接口响应 / 适配后模型 / 页面展示”三层数值一致（允许展示格式差异）。
2. `总收益额` 在所有分支下均满足“纯收益(PnL)”口径。
3. fallback 仅在字段缺失时触发，不覆盖后端已给值。
4. run/stop 后，列表与详情在一次刷新内收敛到一致状态。

## 10. 风险与缓解

1. 风险：测试样本覆盖不足导致误判通过。
- 缓解：固定 4 条以上差异化样本，覆盖有值/缺失/null/状态差异。

2. 风险：页面格式化造成误判（字符串对比失败）。
- 缓解：断言分离“数值语义”与“展示格式”。

3. 风险：全链路失败难定位。
- 缓解：坚持分层执行顺序，E2E 只做烟测，不承载全部口径断言。
