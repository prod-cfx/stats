# 2026-03-20 Quantify Binance Testnet Real Order Account Strategy Validation Design

## 1. 背景与目标

本设计用于一次最小真实链路验收：

- 在 Binance Spot Testnet 以真实行情执行一笔 `BTCUSDT` 市价买入（`10 USDT`）。
- 验证执行结果是否正确映射到「个人中心 -> AI量化」的策略列表与策略详情数据。
- 本轮不评估策略优劣，不扩展策略能力，仅验证“真实下单后账户策略数据是否正确”。

## 2. 范围与非目标

### 2.1 范围

- 交易执行：固定测试链路，单次 `spot market buy`。
- 数据核对：
  - 列表页：`状态 / 收益率 / 今日盈亏 / 更新时间`
  - 详情页：`总收益 / 今日盈亏 / 持仓 / 时间线`
- 后端追溯：`signal -> execution -> position/account-view` 的闭环一致性。

### 2.2 非目标

- 不修改交易策略逻辑。
- 不做前端样式改动。
- 不做全自动 E2E 稳定化建设（本轮只做最小可复验链路）。

## 3. 约束与输入

- 交易所：Binance Spot Testnet。
- 交易对：`BTCUSDT`。
- 下单金额：`10 USDT`。
- 执行方式：复用现有脚本 `fixed-binance:testnet:signal`，带 `--execute`。
- 凭据来源：现有环境配置（不新增硬编码密钥）。

## 4. 方案对比与选型

### 方案 A（选中）

直接使用现有脚本执行真实买入，并通过后端记录 + 前端页面做双向核对。

- 优点：
  - 路径最短，复用现有能力。
  - 变更最少，风险最小。
  - 最符合当前“先验证数据正确性”的目标。
- 缺点：
  - 核对步骤偏手工。

### 方案 B

在方案 A 基础上新增半自动校验脚本（对比接口字段）。

- 优点：后续重复验证效率更高。
- 缺点：需要额外开发临时校验逻辑。

### 方案 C

新增独立最小下单脚本并绕开现有固定链路。

- 优点：脚本逻辑更直观。
- 缺点：重复造轮子，维护成本高，不符合当前仓库收敛方向。

结论：先执行方案 A，通过后再决定是否演进方案 B。

## 5. 端到端数据流设计

1. 触发器（脚本）创建并执行固定测试信号（spot entry buy）。
2. `SignalExecutorService` 执行交易。
3. Trading/Positions/Accounts 写入执行结果与仓位/账本相关数据。
4. Account Strategy View 聚合层读取并计算展示字段。
5. 前端账户策略列表与详情接口渲染聚合结果。

## 6. 执行步骤（Runbook）

1. 前置检查
   - 环境变量齐备（`QUANTIFY_FIXED_BINANCE_TESTNET_*`、`EXCHANGE_ACCOUNT_CRYPTO_KEY`）。
   - fixed testnet seed 上下文存在（固定用户、固定账户、固定策略实例）。
2. 执行真实买入
   - 运行 fixed binance testnet signal 脚本，参数为 `open-spot + execute + 10USDT + BTCUSDT`。
3. 后端核对
   - 核对 signal、execution、positions（或 trades）记录新增且状态成功。
4. 前端核对
   - 列表页与详情页核对目标字段。
5. 输出验收结论
   - 形成“预期值/实际值/数据来源”的字段真值表。

## 7. 通过标准（Acceptance Criteria）

1. 交易执行成功：存在对应成功 execution 记录。
2. 页面展示成功：目标策略在列表页可见，详情页可打开。
3. 字段正确：
   - 列表：`状态/收益率/今日盈亏/更新时间` 非异常。
   - 详情：`总收益/今日盈亏/持仓/时间线` 与买入方向一致。
4. 可追溯：前端展示可追溯到同一笔后端 execution/signal。

## 8. 风险与处置

1. 下单成功但页面不更新
   - 优先排查 account-strategy-view 聚合链路与前端适配层。
2. 页面更新但数值异常
   - 排查 PnL 口径、估值价格来源、日切时区边界。
3. 下单失败
   - 排查 testnet key 权限、symbol 合法性、最小下单金额限制。

## 9. 交付物

- 本设计文档（当前文件）。
- 一次真实买入验证结果（命令输出摘要 + 字段真值表）。

## 10. 后续演进（可选）

- 若本次通过：追加方案 B 半自动校验，降低回归成本。
- 若本次失败：按失败类型分别修复聚合口径或执行链路，再复测。
