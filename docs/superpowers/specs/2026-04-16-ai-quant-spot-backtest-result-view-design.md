# AI Quant 现货/合约回测结果分层展示设计

- 日期：2026-04-16
- 状态：approved-for-spec-review
- 主题：在统一回测结果页内，按 `marketType` 区分现货（spot）与合约（perp）的展示语义

## 1. 背景

当前 AI Quant 回测结果详情页虽然已经能够成功展示现货回测数据，但整体展示心智仍偏向合约语义，导致 spot 用户体验不佳。

典型现象包括：

- 结果页整体像在展示“合约回测”；
- 现货结果中仍容易出现更偏合约语义的理解路径，例如：
  - `entry_long`
  - `totalTrades`
  - `totalOpenTrades`
  - `openPnl`
- 用户需要自己把这些底层字段翻译成“现货买入后仍持有中”的含义。

用户已明确反馈：

- **现货回测结果应该和合约回测结果区分开**；
- 不希望继续让 spot 结果套用 futures/perp 的展示心智；
- 不需要拆成两个完全独立页面，但必须在结果展示层做清晰区分。

## 2. 目标

### 2.1 目标行为

1. **统一结果页骨架保留**，不拆新的路由或完全独立页面；
2. 页面根据 `job.inputSummary.marketType` 切换为：
   - `spot` 展示模式
   - `perp` 展示模式
3. `spot` 结果页强调：
   - 当前持仓
   - 持仓浮盈浮亏
   - 买入建仓 / 卖出平仓
4. `perp` 结果页继续保留现有合约语义：
   - 杠杆
   - long / short
   - 合约交易结果心智
5. 前端允许继续复用同一份底层回测结果数据结构，但必须通过展示适配层转换成用户可理解的语义。

### 2.2 非目标

本次不处理：

- 新建 spot 专属路由或 perp 专属路由；
- 重写回测引擎或修改后端回测结果结构；
- 修改 marker / summary 的底层字段命名；
- 增加新交易指标体系；
- 扩展成“不同 marketType 使用完全不同页面布局”的大改造。

## 3. 方案比较

### 方案 A：只改文案

- 仅把现有页面里的部分词替换成更像现货的话术；
- 不建立独立的结果语义映射层。

**优点**：改动最小。  
**缺点**：只是表面翻译，底层结构和组合关系仍然偏合约，体验改善有限。  
**结论**：拒绝。

### 方案 B：统一骨架 + `marketType` 分层展示（选中）

- 保留一个结果页；
- 基于 `marketType` 决定：
  - 展示哪些模块；
  - 每个模块如何命名；
  - marker / summary / open position 如何映射成面向 spot 或 perp 的表达。

**优点**：
- 明显改善现货用户体验；
- 复用现有页面结构；
- 不引入双路由或大规模重复代码。  
**缺点**：
- 需要梳理展示适配层，而不只是改几个文案。  
**结论**：采用。

### 方案 C：spot / perp 两套完全独立结果页

- 为现货和合约分别维护不同页面或不同大组件树。

**优点**：语义最彻底分离。  
**缺点**：维护成本高，重复逻辑多，当前需求没有必要。  
**结论**：当前阶段拒绝。

## 4. 核心设计决策

### 4.1 页面结构

保留一个统一的回测结果详情页，不新增独立路由。

页面在加载 job / result 后，首先读取：

- `job.inputSummary.marketType`

然后进入：

- `spot` 结果展示模式；
- `perp` 结果展示模式。

也就是说：

- **页面结构统一**；
- **展示语义分层**；
- **marketType 是唯一切换源**。

### 4.2 为什么不拆两套路由

当前 spot 与 perp 回测仍共享：

- job 查询链路；
- result 获取链路；
- equity curve / markers / openPositions 等核心数据结构。

如果为了展示语义差异直接拆成两套页面，会带来：

- 逻辑重复；
- 回归成本增加；
- 未来 spot / perp 共性改动需要同步维护两边。

因此更合理的边界是：

- **一套页面壳层**；
- **一层 marketType-aware 展示适配器**。

## 5. 详细设计

### 5.1 页头与结果上下文

结果页顶部需要明确告诉用户当前查看的结果类型：

- `spot`：显示 **现货回测**
- `perp`：显示 **合约回测**

目的：让用户无需从细节字段中猜测自己看到的结果属于哪一类市场。

### 5.2 结果摘要层

#### spot 模式

对原始结果字段：

- `totalTrades`
- `totalOpenTrades`
- `openPnl`

做展示语义映射：

- `totalTrades` -> **已完成交易**
- `totalOpenTrades` -> **当前持仓**
- `openPnl` -> **持仓浮盈浮亏**

这样像下面这种结果：

- `totalTrades = 0`
- `totalOpenTrades = 1`
- `openPnl > 0`

对用户会直接表现成：

- 已完成交易：0
- 当前持仓：1
- 持仓浮盈浮亏：+X

而不是要求用户自己理解“为什么总交易数是 0，但还有 open trades”。

#### perp 模式

保留现有更偏合约的摘要语义，不强制改名。

### 5.3 Marker 展示层

底层 marker kind 不要求修改，但前端展示标签按 `marketType` 转换。

#### spot 模式

- `entry_long` -> **买入建仓**
- `exit_long` -> **卖出平仓**

spot 页面不直接把 `long` 暴露给用户作为主要表达。

#### perp 模式

继续保留：

- `entry_long`
- `entry_short`
- `exit_long`
- `exit_short`

对应的 long / short 合约语义。

### 5.4 持仓明细层

#### spot 模式

`openPositions` 模块应强调：

- 当前持仓数量
- 持仓均价
- 当前浮盈浮亏

并避免使用：

- 未平仓合约
- long position
- 杠杆仓位

#### perp 模式

继续保留现有合约仓位表达。

### 5.5 symbol 展示层

#### spot 模式

像 `BTCUSDT:SPOT` 这种内部编码不应直接原样暴露给用户。

建议前端展示为：

- `BTCUSDT 现货`
或
- `BTC/USDT（现货）`

#### perp 模式

可展示为：

- `BTCUSDT 合约`
或沿用现有合约符号表达。

### 5.6 需要弱化或隐藏的 spot 语义

spot 页面应弱化或隐藏以下合约优先心智：

- 杠杆
- 保证金
- 强平 / 爆仓联想
- long / short 作为主要用户术语

这并不要求删除底层字段，只要求不在 spot 页面把它们作为中心展示语义。

## 6. 展示适配层边界

本次建议在前端结果页中引入一个明确的展示适配层，职责如下：

1. 输入：
   - job summary
   - result payload
   - marketType
2. 输出：
   - 页头模式标签
   - 摘要卡片展示字段
   - marker 展示标签
   - openPositions 展示标签
   - symbol 用户态展示文案

这样可以保证：

- 底层 API 数据结构不被 UI 逻辑污染；
- spot / perp 差异集中在一个层次，而不是分散在各个 JSX 分支里；
- 后续如果再新增 marketType 相关展示差异，也有明确扩展点。

## 7. 验收标准

1. `marketType=spot` 时，结果页明确显示 **现货回测**；
2. `marketType=perp` 时，结果页明确显示 **合约回测**；
3. spot 模式下：
   - `entry_long` 被展示为 **买入建仓**；
   - `exit_long` 被展示为 **卖出平仓**；
4. spot 模式下：
   - `totalTrades` 被展示为 **已完成交易**；
   - `totalOpenTrades` 被展示为 **当前持仓**；
   - `openPnl` 被展示为 **持仓浮盈浮亏**；
5. spot 模式下不再突出杠杆 / 保证金 / long-short 合约心智；
6. perp 模式下现有合约结果表达保持正常；
7. 同一结果页组件在 spot / perp 两种模式下都能正确渲染，不出现字段错位、空文案或混合语义。

## 8. 测试策略

### 8.1 展示适配层测试

至少覆盖：

- 输入 `marketType=spot` 时，摘要 / marker / symbol / open position 的标签映射正确；
- 输入 `marketType=perp` 时，保持现有合约语义；
- 未知或缺失 `marketType` 时，使用安全兜底（默认按现有逻辑或显式 fallback）。

### 8.2 结果页组件测试

至少覆盖：

- spot 结果页显示“现货回测”；
- perp 结果页显示“合约回测”；
- spot 结果页中 `totalTrades=0 + totalOpenTrades=1 + openPnl>0` 能正确渲染成“已完成交易 / 当前持仓 / 持仓浮盈浮亏”；
- marker 在两种模式下使用不同标签。

### 8.3 回归测试

确保：

- 现有结果页查询链路不受影响；
- equity curve / trades / markers / openPositions 的原始数据不被错误篡改；
- 合约结果页不会被现货文案污染。

## 9. 风险与约束

### 风险 1：只改局部文案，未建立展示适配层

这会导致 spot / perp 规则散落在多个组件里，后续非常容易继续混淆。

**约束**：必须有清晰的展示语义映射边界。

### 风险 2：过度拆分成双页面

这会让现有结果页的共性逻辑被复制，带来维护成本与回归负担。

**约束**：当前阶段坚持单页骨架，不拆双路由。

### 风险 3：把底层结果字段改成 UI 专用字段

这会污染后端 / 通用结果模型的职责边界。

**约束**：尽量通过前端适配层转换，不重写后端基础结果结构。

## 10. 最终结论

本次采用：

- **统一回测结果页骨架**
- **按 `marketType` 分层展示 spot / perp 语义**
- **通过展示适配层把底层结果字段映射成用户可理解的表达**

最终目标是：

- spot 用户看到的是“现货回测、当前持仓、持仓浮盈浮亏、买入建仓”；
- perp 用户看到的是“合约回测、杠杆、long/short 语义”；
- 页面结构统一，但认知体验准确分层。
