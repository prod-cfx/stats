# Backend E2E 并行回归与修复设计

## 1. 背景与目标

针对 `apps/backend/e2e` 下全部 E2E 文件执行一次并行回归，执行约束如下：

- 使用 `dx` 按文件执行（单次只指定一个文件）。
- 允许并行，固定并发度 `4`。
- 目标是“真实全量跑通”，不做范围收缩。
- 对失败用例需要做闭环处理：
  - 若测试代码过时，允许删除或修改测试；
  - 若是业务实现缺陷，修复代码。

## 2. 范围与非目标

### 2.1 范围

- 目录：`apps/backend/e2e/*.e2e-spec.ts`
- 执行方式：`dx test e2e backend <file>`
- 输出：完整回归结果与修复清单

### 2.2 非目标

- 不做与失败无关的大规模重构
- 不新增与当前失败无关的功能
- 不改变全局测试框架配置（除非被验证为根因）

## 3. 方案对比与选型

### 3.1 方案 A（选中）：并行执行 + 失败即修复

流程：4 并发跑单文件 -> 失败即分类与修复 -> 受影响文件重跑 -> 最终全量确认。

优点：最快形成闭环，减少“先盘点后修复”的二次定位成本。
缺点：上下文切换较频繁。

### 3.2 方案 B：先盘点再批量修复

优点：全局视图清晰。
缺点：修复后返工与二次回归成本高。

### 3.3 方案 C：按模块批次回归

优点：模块隔离更好。
缺点：组织开销较高，对当前“先跑通”目标不最优。

结论：采用方案 A。

## 4. 执行架构

### 4.1 任务发现

- 收集 `apps/backend/e2e` 下所有 `*.e2e-spec.ts` 文件。
- 生成待执行队列与结果状态表。

### 4.2 调度策略

- 固定并发度：`4`
- 每个任务独立执行：`dx test e2e backend <file>`
- 每个文件记录独立日志与退出码。

### 4.3 结果状态

- `PASS`：首次成功。
- `FAIL`：执行失败，进入根因分类。
- `FLAKY`：首次失败后重跑通过，需二次确认。

### 4.4 重试与隔离

- 首次失败后立即重跑 1 次。
- 若出现 `FLAKY`，要求连续 2 次通过才可归档为通过。
- 对疑似共享资源冲突的文件，降级为串行确认。

## 5. 失败分类与修复准则

### 5.1 `FAIL-TEST-STALE`（测试过时）

判定条件：

- 断言依赖旧契约/旧字段；
- 测试场景已被业务废弃；
- 用例不再代表有效用户路径。

处理策略：

- 优先更新测试断言与 fixture；
- 若无业务价值且确属过时，允许删除用例；
- 删除需在回归报告注明原因标签：
  - `obsolete-flow`
  - `replaced-by-new-contract`
  - `external-nondeterministic-no-business-value`

### 5.2 `FAIL-CODE-BUG`（实现缺陷）

判定条件：

- 当前业务契约应当成立，但实现偏离导致失败。

处理策略：

- 仅做最小必要修复；
- 禁止顺带无关重构；
- 修复后必须重跑对应文件并检查连带影响。

### 5.3 `FAIL-INFRA`（环境/依赖阻塞）

判定条件：

- 明确属于环境级问题（如 DB schema 缺失、外部依赖不可用）。

处理策略：

- 先输出可复现证据与最小修复动作；
- 修复环境后继续执行剩余队列。

## 6. 验证闭环

每个修复项执行以下验证链：

1. 重跑受影响单文件；
2. 重跑同模块关键文件（若存在）；
3. 全量回归 `apps/backend/e2e`。

完成标准：

- 全量通过；
- 无未解释失败；
- 所有删除/修改项均有明确理由。

## 7. 交付物

- 回归结果表（每个 E2E 文件状态：`PASS/FLAKY/FAIL`）
- 失败根因分类清单
- 修复清单（测试修改、测试删除、代码修复）
- 最终验证结果（分层重跑 + 全量重跑）

## 8. 风险与控制

- 外部接口波动导致假失败：使用“失败重跑 + 串行确认”降低误判。
- 并发下数据冲突：固定并发 4，并对冲突项串行复验。
- 误删测试：仅删除“无业务价值且过时”用例，并强制记录删除理由。
